# Protocolo de incidencias

## Objetivo

Un protocolo de incidencias define qué pasos debe seguir el técnico cuando un usuario reporta un problema.

Sirve para no improvisar y para que todas las incidencias se gestionen con el mismo criterio.

## Flujo general

1. Recibir incidencia.
2. Registrar datos mínimos.
3. Valorar si se puede resolver remotamente.
4. Ejecutar solución remota si es posible.
5. Comprobar si queda resuelta.
6. Si no se puede resolver remotamente, desplazarse al equipo.
7. Revisar si se puede solucionar en el lugar.
8. Si no se puede, sustituir equipo o llevarlo a taller.
9. Comprobar garantía.
10. Recuperar datos si es necesario.
11. Reparar, sustituir o comprar piezas.
12. Entregar equipo o datos al usuario.
13. Informar al usuario.
14. Registrar datos de resolución.

## Preguntas clave

### ¿Se puede solucionar remotamente?

Si sí, usar herramientas como:

- SSH
- RDP
- VNC
- asistencia remota

### ¿Se puede solucionar en el sitio?

Si no es remoto, revisar físicamente el equipo.

Ejemplos:

- cable desconectado
- periférico dañado
- pantalla sin alimentación
- error de red local

### ¿Tiene garantía?

Si hay garantía, no conviene abrir o manipular más de lo necesario.

Se debe reclamar reparación o recambio.

### ¿Hay datos que recuperar?

Antes de reinstalar, formatear o sustituir disco, preguntar si hay datos importantes.

## Cierre de incidencia

Una incidencia no se cierra solo porque “ya funciona”.

Debe quedar registrado:

- qué se hizo
- si se recuperaron datos
- si se sustituyó equipo
- si se compraron piezas
- si el usuario fue informado
- fecha de cierre

## Buena práctica

El técnico debe evitar crear otro problema mientras arregla el primero. Antes de tocar datos, discos o perfiles, se debe pensar en copia de seguridad o recuperación.
